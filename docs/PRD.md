# PRD: 포이즌 리셀 자동화 시스템

> 버전: 0.2 (업데이트)

> 작성일: 2026-03-24  
> 작성자: 문지영

---

## 1. 개요 (Overview)

포이즌(POIZON) Open Platform API를 활용하여, 리셀러가 **최적의 상품을 선택하고 최저가 기준으로 입찰을 자동화**할 수 있는 웹 기반 관리 도구.

---

## 2. 핵심 문제 (Problem Statement)

리셀러가 수익성 있는 입찰을 하려면:
1. 포이즌에서 현재 입찰 가능한 상품/가격 정보를 확인해야 함
2. 동시에 네이버 가격비교에서 해당 상품의 시중 최저가(매입원가)를 확인해야 함
3. 이 두 가지 데이터를 비교해 **수익이 남는 상품에만 입찰**해야 함

현재는 이 과정을 수동으로 하기 때문에 비효율적이고 느림.

---

## 3. 목표 (Goal)

- 포이즌 입찰 정보와 네이버 최저가를 **한 화면에서 비교**할 수 있게 함
- 리스팅 선택 → 가격 분석 → 입찰까지의 과정을 **최소한의 클릭**으로 처리
- 수익성 분석을 자동화하여 **좋은 기회를 놓치지 않음**

### 📌 개발 단계 (Phases)
- **Phase 1**: UI 구축 및 Poizon Open Platform API 완벽 연동 (상품 검색, 리스팅, 입찰 기능 중심)
- **Phase 2**: 네이버 쇼핑 API 연동 및 수익성 자동 계산(마진 분석표) 기능 결합
- **Phase 3**: 후속 시스템 고도화 (입찰 알림, 재입찰 로직 등)

---

## 4. 사용자 워크플로우 (User Flow)

```
[포이즌 API]
    ↓ 입찰 가능 상품 목록 조회
[리스팅 화면]
    ↓ 관심 품번 선택
[상품 분석 화면]
    ↓ 네이버 최저가 자동 비교
[수익성 계산]  ← 포이즌 입찰가 vs 네이버 최저가
    ↓ 수익 있는 상품 확인
[입찰 실행]  ← 포이즌 API로 자동 입찰
```

## 5. 핵심 기능 (Core Features)

### 5.1 입찰 정보 대시보드
- 포이즌 API에서 현재 입찰 가능한 상품 목록 실시간 대량 조회 (품번 직접 검색 및 브랜드명 서치)
- 상품명, 품번, 현재 입찰가 정보와 **중국 시장 최근 30일 판매량 통계** 병합 표시
- **[UI/UX]** 각 상품(SPU) 행의 펼침(Chevron) 아이콘 클릭 시 하위 옵션(SKU별 색상, 사이즈, 바코드, SKU 단위 판매량/입찰가) 정보를 볼 수 있는 아코디언 구조 지원
- 필터/정렬: 브랜드, 카테고리, **거래량순(통계 API 기반 자체 정렬)**, 가격대

### 5.2 품번 선택 & 분석
- 관심 품번을 체크박스로 선택
- 선택된 품번에 대해 **네이버 쇼핑 최저가 자동 검색**
  - ⚠️ **데이터베이스 기반 몰 필터링**: 관리자 화면에서 가격비교에 포함/제외할 메이저 종합몰 목록을 DB(White/Black 리스트)로 관리하여 동적 적용
  - 가격 낮은 순 정렬 후 상위 결과를 매입원가 기준으로 채택
- 포이즌 입찰가 vs 네이버 최저가 비교표 자동 생성
- 예상 수익(마진) 자동 계산

### 5.3 수익성 분석

**수수료 계산 로직 (확정)**:
```
기본 수수료 = 판매가 × 10%
실제 수수료 = max(15,000원, min(기본 수수료, 45,000원))
예상 수익 = 네이버 최저가 - 포이즌 입찰가 - 실제 수수료
```

| 판매가 | 기본 10% | 적용 수수료 |
|---|---|---|
| 100,000원 | 10,000원 | **15,000원** (최소) |
| 200,000원 | 20,000원 | **20,000원** |
| 450,000원 | 45,000원 | **45,000원** |
| 600,000원 | 60,000원 | **45,000원** (최대) |

- 수익률(%)로 정렬/필터링
- 손익분기점 기준으로 색상 표시 (✅ 수익 / ❌ 손해)

### 5.4 입찰 실행 (최대 자동화)
- 포이즌 API 제공 기능 내에서 **최대한 자동화**:
  - **자동 입찰(Automatic Bidding)**: API의 자동 입찰 기능 적극 활용
  - **스마트 리스팅(Smart Listing)**: 최적 가격으로 자동 등록
  - **가격 자동 갱신**: 경쟁 입찰가 변동 시 자동 재설정
- 입찰 조건 설정: 수익률 X% 이상인 경우만 자동 입찰
- 입찰 이력 저장 및 관리

## 8. 설계 및 개발 원칙 (Development Principles)

### 8.1 API 연동 사전 검증 워크플로우 (Preventive Rule)
외부 API(특히 Poizon 오픈 플랫폼) 연동 시 공식 문서 명세와 실제 반환 데이터가 상이하여 발생하는 프론트엔드 오류를 원천 차단하기 위해 다음 원칙을 준수함:
1. **공식 문서 완전 해부**: 연동 전 `browser_subagent` 등을 활용해 공식 문서의 'Response Sample JSON'을 스크래핑하고 완벽한 Type/Zod Schema(`types/poizon.ts` 등)를 선언함.
2. **독립 샌드박스 교차 검증 (TDD 방식)**: UI 로직을 작성하기 전, `api/test/` 등 백엔드 단에서 격리된 테스트 라우트를 구성해 실제 라이브 데이터를 한번 호출(Dump)해봄.
3. **타입 일치 여부 확인 후 UI 렌더링**: 실제 반환된 JSON Key 구조(예: `salesAmount` ➡ `commoditySales.globalSoldNum30` 등 숨겨진 변수명)가 정의한 타입과 1:1로 일치하는지 눈으로 먼저 1차 점검 후 프론트엔드 로직에 반영함.

### 8.2 식별자(ID) 타입 캐스팅 엄수 (Type-Casting Rule)
포이즌과 같은 리거시 외부 API는 동일한 식별자(예: `spuId`, `skuId`)임에도 엔드포인트에 따라 숫자(Number)와 문자열(String)을 혼용하여 반환할 위험이 상존함.
- 자바스크립트의 내장 `Map`, `Set` 객체나 `===` 연산 환경에서는 타입 불일치 시 치명적인 데이터 병합(Merge) 누락 버그로 직결됨.
- **방지 대책**: 앞으로 외부 API에서 받아온 고유 식별계(ID) 데이터를 프론트엔드나 로컬 구조체에서 순회·매핑(Mapping)할 시, **반드시 `Number(id)` 혹은 `String(id)`으로 명시적 강제 형변환(Type-Casting)** 을 우선 거친 뒤 비즈니스 로직에 투입하도록 코드 컨벤션으로 강제함.
